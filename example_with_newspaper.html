Phase Analysis

Phase 1: Build fast, fix in prod

The team wrote 40 specs in 14 active days and shipped 23 PRs. The fix profile is dominated by a single spec: INF-01 (Azure deployment) accounts for 16 of 28 fixes (57%). These were direct pushes to main (region mismatches, SASL credentials, load balancer timeouts, Cosmos init containers). All first-contact-with-Azure problems that no spec could have predicted. The 32 feature specs generated only 2 follow-up fixes total. Feature spec quality was high from day one.

Phase 2: More specs, more infra surface area

The spec mix flipped: 54% TR (technical refactoring) vs 34% FR. The team was hardening the platform: multi-portal monorepo, topic consolidation, GraalVM, OPA auth, K6 load testing. INF-01 continued bleeding (10 more fixes), INF-04 appeared as a new fix source (7 fixes for portal subpath routing), and FR-AU-01 (authentication) generated 4 fixes as OAuth touched every portal. But the per-PR fix rate improved (0.95 vs 1.22 in Phase 1) despite the higher absolute count. The team was merging more PRs with proportionally fewer fixes per PR.

Phase 3: v0.0.3 — Highest throughput, lowest fix rate per PR

Phase 3 was the most productive: 250 commits, 46 merged PRs, and 23 specs in 19 days. The fix-per-PR ratio hit its lowest at 0.28. The process matured significantly: 14 of 23 fix-related changes went through PRs (vs only 1 of 27 in Phase 1). Most direct-push fixes were dashboard cosmetics or dependency compatibility patches, not spec failures. New capabilities shipped cleanly: scenario modeling, Delta Lake, Gateway API migration, LLP cycle tracking, parts provenance, and Terraform infrastructure all landed with minimal follow-up fixes.

Current Iteration: Onboarding Portal — Domain pivot, near-zero fixes

The team pivoted entirely to the onboarding portal: 213 commits, 38 merged PRs, and 30 specs in 22 days. The fix-per-PR ratio dropped to 0.13 — an all-time low. Only 1 direct-push fix landed (a dashboard date string). All 4 fix PRs traced back to backlog specs (stepper state management, SWA lockfile sync, shared library build warnings) — none were feature regressions. The specs-as-backlog pattern dominated: Greg authored the multi-stage form scaffold and OB workflow specs, while Brett, Casey, and Kellee picked them up for implementation. 83% of specs are feature work.

FROM ZERO TO SIX DOMAINS IN FIVE WEEKS

Federated GraphQL architecture enables independent domain evolution
Architecture Desk

The system rose from an empty repository to six fully operational domain services: Aircraft, Maintenance, Facility, Fleet, Parts, and Vendors. Each runs as an independent Quarkus subgraph behind an Apollo Router federation gateway, with its own Cosmos DB database.

A dual EventHub emulator pattern handles 19 Avro event types across five consolidated topics, with cross-domain references maintaining eventual consistency through a fully meshed subscription graph.

Each domain owns its Cosmos DB database and GraphQL schema, communicating only through Avro-serialized events. The pattern enforces true bounded contexts — no shared databases, no synchronous coupling, no shortcuts.

Fig. 1: An architect, mid-epiphanyThe frontend evolved from a single React application into six specialized portals with a shared component library, each deployable independently via nginx subpath routing on AKS. The procurement portal ships as a Progressive Web App with offline capability.

Open Policy Agent enforces role-based access at the gateway, while an immutable audit trail captures every mutation through Cosmos change feed functions writing to Azure Table Storage. Observability spans the full stack with OpenTelemetry, Prometheus, and Grafana dashboards tracking request latency, error rates, and domain health.
